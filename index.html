SQL Injection (SQLi) - OWASP #1

Critical Risk

What is SQL Injection?

SQL Injection is a code injection technique that might destroy your database. It is one of the most common web hacking techniques.

Impact Level:
90% - Critical

How it works:

SQL
// Normal query
SELECT * FROM users WHERE username = 'admin' AND password = 'password';

// Injected query (classic 1=1 bypass)
SELECT * FROM users WHERE username = 'admin' OR '1'='1' --' AND password = '';

Common Attack Vectors:

Type Payload Example Effect
Union-based ' UNION SELECT null, username, password FROM users-- Extracts data from other tables
Error-based ' AND 1=CAST((SELECT @@version) AS INT)-- Extracts info via error messages
Blind SQLi ' AND SLEEP(5)-- Time-based inference attacks
Boolean-based ' AND 1=1-- vs ' AND 1=2-- Inference through true/false responses

Prevention Techniques:

Parameterized Queries

Use prepared statements with parameterized queries to separate SQL code from data

Input Validation

Whitelist allowed characters and validate all user inputs

Stored Procedures

Use stored procedures with strict parameter definitions

ORM Libraries

Use Object-Relational Mapping libraries that handle SQL safely

Cross-Site Scripting (XSS) - OWASP #7

High Risk

What is XSS?

XSS enables attackers to inject client-side scripts into web pages viewed by other users, potentially stealing data or performing actions on behalf of users.

XSS Evolution Timeline:

1999
First XSS vulnerability discovered
2005
Samy worm spreads via MySpace XSS
2010
Content Security Policy (CSP) introduced
Present
XSS remains in OWASP Top 10

XSS Types Comparison:

Type How it Works Example Impact
Reflected XSS Script reflected from current request ?search=<script>alert(1)</script> Medium
Stored XSS Script stored on server (database) Malicious comment stored High
DOM-based XSS Client-side script manipulation document.write(userInput) Medium

Modern XSS Prevention:

  • Content Security Policy (CSP): Define allowed script sources
  • Input Validation: Whitelist allowed HTML tags and attributes
  • Output Encoding: Encode data based on context (HTML, JS, CSS, URL)
  • HttpOnly Cookies: Prevent JavaScript cookie access
  • Regular Security Testing: Use automated scanners and manual testing

Cross-Site Request Forgery (CSRF)

Medium Risk

What is CSRF?

CSRF tricks a user's browser into executing unwanted actions on a trusted site where they're authenticated.

Example attack:

HTML
<!-- Malicious website content -->
<img src="http://bank.com/transfer?to=attacker&amount=1000">

Prevention:

  • CSRF tokens in forms
  • SameSite cookie attribute
  • Check Origin/Referer headers
  • Require re-authentication for sensitive actions

Directory Traversal

Low Risk

What is Directory Traversal?

Allows attackers to access files and directories outside the web root directory.

Example:

HTTP Request
Normal request:
GET /view?file=image.jpg

Attack request:
GET /view?file=../../../../etc/passwd

Prevention:

  • Input validation (whitelist allowed files)
  • Use basename() to remove path components
  • Store files outside web root
  • Proper file permissions

Security Headers

Information

Important Security Headers:

Content-Security-Policy (CSP)

Prevents XSS by specifying allowed content sources

Content-Security-Policy: default-src 'self'
X-Frame-Options

Prevents clickjacking by controlling frame embedding

X-Frame-Options: DENY
X-Content-Type-Options

Prevents MIME type sniffing

X-Content-Type-Options: nosniff
Strict-Transport-Security (HSTS)

Forces HTTPS connections

Strict-Transport-Security: max-age=31536000
